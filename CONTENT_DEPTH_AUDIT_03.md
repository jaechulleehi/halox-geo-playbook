## 03장 재보강 감사 메모: Fan-out 개념 분리

### 문제 진단

로이드 피드백처럼 기존 03장은 `fan-out`을 사람이 질문을 많이 펼치는 작업처럼 설명한 부분이 많았습니다. 하지만 HaloX에서 중요한 fan-out은 단순 질문맵이 아니라, 사용자의 한 질문을 AI가 답변하기 위해 내부적으로 여러 하위 질문과 검색/검증 작업으로 나누는 현상입니다.

따라서 이번 재보강의 핵심은 세 개를 분리하는 것입니다.

| 구분 | 의미 | 03장에서 맡을 역할 |
|---|---|---|
| 사용자 질문 확장 | 우리가 키워드, PAA, 연관 검색, 고객 질문으로 질문셋을 넓히는 작업 | 기준선 측정용 프롬프트 풀 만들기 |
| AI query fan-out | AI가 한 질문을 답하기 위해 내부적으로 정의/비교/검증/source/URL 확인 같은 하위 작업으로 쪼개는 현상 | GPT/AI 검색 패턴을 추정하고 콘텐츠 갭을 찾는 핵심 개념 |
| 검색 단서 | Google PAA, 연관 검색, 자동완성, SERP 상위 문서에서 얻는 공개 신호 | 사용자 질문 확장의 재료이자 AI fan-out 추정의 보조 단서 |

### 반영한 소스

| 소스 | 반영 내용 |
|---|---|
| HaloX Glossary `쿼리 팬아웃` | 하나의 질문을 여러 하위 쿼리로 나눠 검색 후 합성한다는 정의 반영 |
| HaloX Glossary `PAA` | People Also Ask는 사용자 질문 확장의 단서이지 AI fan-out 자체는 아니라는 구분 반영 |
| Obsidian `기능 아이디어 - fan out 질문맵 기반 콘텐츠 기획 뷰` | 사용자의 단일 질문보다 모델 내부 하위 질문 구조가 중요하다는 문제 정의 반영 |
| Obsidian `기능 설계 - fan out 질문맵 기반 콘텐츠 기획 뷰` | 질문 노드, 커버 상태, gap_note, next_action 구조 반영 |
| Obsidian `인용 사유 fan out 사례집 아카이브 뷰` | 왜 인용됐는지, 어떤 하위 질문으로 확장됐는지 사례화해야 한다는 관점 반영 |
| `SOURCE_MAP.md` | fan-out 질문맵, 질문셋 포션/커버리지, AI 설명 품질 진단 연결 반영 |
| Google AI features | AI 기능이 웹 콘텐츠를 답변에 활용할 수 있다는 공식 맥락 연결 |
| Google helpful content / SEO Starter Guide | 질문 수보다 독자에게 실제 도움이 되는 답변 재료와 발견 가능성 기준 반영 |

### 보강 방향

- 03 parent: fan-out을 `AI가 내부에서 확장하는 질문 패턴`으로 재정의.
- 03-01: fan-out과 사용자 질문 확장을 분리하고, AI가 수행할 수 있는 하위 작업을 `정의/비교/검증/source/URL/실행/리스크`로 설명.
- 03-02: 질문 포트폴리오를 단순 유형표가 아니라 `측정용 사용자 질문셋`과 `AI 내부 fan-out 패턴`의 분리 기준으로 재작성.
- 03-03: 콘텐츠 갭을 사람이 만든 질문 목록 기준이 아니라 AI의 하위 판단 작업을 막는 빈칸으로 재정의.

### 핵심 모델

```text
SEO/PAA/연관검색/고객 질문
→ 우리가 측정할 사용자 질문셋
→ AI가 답변 중 수행하는 query fan-out 추정
→ fan-out 노드별 source/URL/콘텐츠 커버리지 확인
→ 콘텐츠 갭/source 갭/기술 갭/메시지 갭 분리
→ 04장 콘텐츠 구조, 05장 출처 전략, 06장 기술 점검으로 연결
```
