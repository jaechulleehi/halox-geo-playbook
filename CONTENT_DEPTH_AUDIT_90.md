# 90장 산업별 GEO 케이스북 점검 메모

## 결론

90장은 현재 `업종별 사례 목록`의 형태는 갖췄지만, 로이드가 말한 기준으로 보면 아직 얕다. 각 페이지가 60~80줄 수준이고, 대부분 `상황 → SEO 키워드 → AI 질문 → 막히는 지점 → 실행 순서 → 좋은/나쁜 예`라는 같은 골격을 반복한다. 케이스북이라면 독자가 자기 업종에 바로 가져갈 수 있는 `상황별 판단표`, `Before/After`, `질문셋 예시`, `URL/출처/기술/리뷰/상품 데이터 액션표`, `리포트 샘플`, `30일 실행표`가 더 필요하다.

90장은 07장/11장/12장의 요약판이 아니라, 앞 장의 개념을 실제 업종 문제로 바꾸는 사례형 실행 장이어야 한다.

## 현재 구조 진단

| 페이지 | 현재 상태 | 부족한 점 | 우선 보강 방향 |
|---|---|---|---|
| 90 parent | 케이스북 사용법과 페이지 목록만 있음 | 업종을 나누는 실무 기준이 약함 | `의사결정자 / 답변 단위 / 신뢰 출처 / 전환 행동 / 실패 리스크` 기준표 추가 |
| 90-01 PR/마케팅 에이전시 | GEO 서비스 설명 구조만 있음 | 상품화/제안/리포트 샘플이 약함 | 진단형/전략형/운영형 상품 구조, 1페이지 제안서/리포트 예시 추가 |
| 90-02 캠페인 URL | 반복 인용 개념은 있음 | URL 생애주기와 측정 타임라인이 약함 | D-7/D-Day/D+7/D+14/D+30 측정표, URL별 역할/redirect/canonical 처리 추가 |

| 90-04 로컬/전문 서비스 | NAP/지도/리뷰 언급은 있음 | 지점/시술/상담/예약 전환까지의 실제 운영표가 부족함 | 지역 질문셋, 플레이스 설명, 리뷰 응답, 방문 전환 체크리스트 추가 |
| 90-05 금융/규제 산업 | 리스크와 신뢰 방향은 있음 | 안전/위험 표현, 규제형 질문 대응, 공식 출처 맵이 약함 | 위험 문구/안전 문구 비교, 신뢰 출처 우선순위, 비브랜드 질문 액션표 추가 |
| 90-06 커머스/플랫폼 | Product schema/feed 관점은 있음 | 11장과 차별화되는 케이스 운영 흐름이 약함 | 상품 데이터 불일치 사례, AI 구매 질문별 데이터 액션, feed/schema/상세페이지 QA표 추가 |

## 왜 얕게 느껴지는가

1. 각 페이지가 같은 형식을 반복하지만 업종별 고유한 실패 장면이 짧다.
2. `좋은 예/나쁜 예`가 한 줄 수준이라 실제 Before/After처럼 보이지 않는다.
3. 내부 백로그에 있는 실제 소스 축이 본문으로 충분히 전환되지 않았다.
4. 07장/11장/12장과 연결은 있지만, 90장이 맡는 `케이스 실행` 역할이 독립적으로 강하지 않다.
5. HaloX 블로그/제품 프레임이 링크로만 붙어 있고, 사례 속 판단 기준으로 녹아든 부분이 적다.

## 보강 기준

90장을 보강할 때는 각 페이지마다 아래 블록을 넣는 것이 좋다.

| 블록 | 역할 |
|---|---|
| 실제 상황 | 어떤 조직/팀이 어떤 문제로 GEO를 보게 되는지 |
| 첫 실패 | 왜 기존 SEO/PR/콘텐츠 방식만으로는 부족했는지 |
| 질문셋 | 브랜드/비브랜드/비교/검증/전환 질문 예시 |
| 자산맵 | owned/earned/third-party/technical asset 구분 |
| Before/After | 약한 설명과 개선된 설명 비교 |
| 실행표 | 2주 또는 30일 단위 액션, 담당, 완료 기준 |
| 측정표 | mention/source/citation/경쟁사/재측정 기준 |
| HaloX 연결 | 어떤 리포트/지표/블로그가 이 사례를 검증하는지 |

## 페이지별 세부 보강안

### 90-01 PR/마케팅 에이전시

- `GEO를 어떻게 팔까`가 아니라 `현재 AI 답변 상태를 어떻게 보여줄까`로 시작한다.
- 진단형/전략형/운영형/교육형 패키지 구분표가 필요하다.
- 제안서 1페이지 구조: 현재 상태, 빠진 질문군, 반복 인용 출처, 30일 액션, 재측정 기준.
- PR 보도자료 수와 AI citation 안정성을 구분하는 표가 필요하다.

### 90-02 캠페인 URL

- 캠페인 URL을 `대표 랜딩 / 보도자료 / 인플루언서 글 / 리뷰 / UTM 추적 URL / 종료 URL`로 나눠야 한다.
- D-7/D-Day/D+7/D+14/D+30 측정 타임라인이 필요하다.
- 단기 노출, 반복 source, 화면 citation, 검색 TOP10 교차율을 분리해야 한다.
- 캠페인 종료 후 redirect/canonical/아카이브 처리 기준을 넣어야 한다.

### 90-03 엔터프라이즈 뉴스룸

- 뉴스룸을 `콘텐츠 저장소`가 아니라 `엔티티 허브`로 보는 기준을 더 강하게 잡아야 한다.
- 회사 팩트시트, 리더십 바이오, 제품/서비스 허브, 이슈 FAQ, 미디어 키트, 외부 기사 맵을 분리해야 한다.
- 보도자료 Before/After가 필요하다: 날짜순 보도자료 → Answer-first 요약/핵심 숫자/FAQ/관련 링크/공식 입장.
- 05장 오프사이트 엔티티와 06장 Organization/Person/ProfilePage schema를 연결해야 한다.

### 90-04 로컬/전문 서비스

- 12장 요약이 아니라 지점/전문 서비스 케이스 운영표가 되어야 한다.
- 지역+서비스, 지역+상황, 비용, 후기, 리스크, 예약 전환 질문을 실제 표로 넣는다.
- NAP 표준화 순서, 플레이스 설명 문장, 리뷰 응답 원칙, 방문 전 정보(주차/시간/예약/준비물)를 분리한다.
- 병원/법률/미용/교육처럼 표현 리스크가 큰 업종의 안전 문구 예시가 필요하다.

### 90-05 금융/규제 산업

- 비브랜드 질문에서 전환이 생기는 구조를 더 구체화해야 한다.
- `추천/비교/수수료/리스크/가입 조건/보안/세금/규제` 질문 유형별 액션표가 필요하다.
- 위험 표현과 안전 표현을 나눠야 한다. 예: `수익 보장` 대신 `조건/리스크/수수료를 확인해야 한다`.
- 공식 FAQ, 약관, 수수료표, 공시/기관 자료, 고객보호 페이지를 source 우선순위로 정리해야 한다.

### 90-06 커머스/플랫폼

- 11장 심화와 겹치지 않게 `실제 운영 케이스` 중심으로 바꿔야 한다.
- 상품 상세페이지, Product schema, merchant feed, 리뷰, 정책 페이지의 불일치 사례가 필요하다.
- 추천/비교/조건/정책/신뢰 질문별로 필요한 데이터 필드를 표로 넣는다.
- 품절/옵션/가격/배송/반품/리뷰가 서로 다를 때 AI 구매 에이전트가 어떻게 오해할 수 있는지 설명해야 한다.

## 작업 순서 제안

1. 90 parent를 먼저 보강해 케이스북의 읽는 기준을 잡는다.
2. 90-01~90-03을 PR/캠페인/뉴스룸 묶음으로 1차 보강한다.
3. 90-04~90-06을 로컬/규제/커머스 묶음으로 2차 보강한다.
4. 각 페이지에 Before/After와 30일 실행표를 넣는다.
5. 마지막에 07장/11장/12장/05장/06장 링크 정합성을 다시 본다.

## 바로 반영할 수 있는 최소 기준

- 모든 child page에 `이 케이스의 첫 실패` 섹션 추가.
- 모든 child page에 `질문셋 → 자산 → 액션 → 측정` 표 추가.
- 최소 3개 페이지에는 실제 Before/After 예시 추가: 90-01, 90-03, 90-06.
- 90-02에는 D-7~D+30 타임라인 추가.
- 90-04/90-05에는 안전 표현/리뷰 응답/규제 리스크 예시 추가.

## 결론

90장은 보강할 가치가 크다. 지금 구조는 케이스북의 목차 역할은 하지만, 아직 `실제 사례를 읽고 바로 따라 할 수 있는 장`은 아니다. 우선 parent와 90-01~90-03을 먼저 깊게 만든 뒤, 90-04~90-06을 이어서 보강하는 순서가 가장 안전하다.
