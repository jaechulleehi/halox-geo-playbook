## FAQ/표/schema는 언제 쓰는가

FAQ, 표, 리스트, schema는 글을 꾸미는 장식이 아니라 질문과 답을 구조화하는 도구입니다. 특히 비교 기준과 실행 기준은 표로 분리할수록 AI가 이해하기 쉽습니다.

본문이 긴 문단으로만 되어 있으면 AI가 항목 간 차이를 파악하기 어렵습니다. AI가 HTML을 Markdown에 가까운 텍스트 구조로 바꿔 읽는다고 생각하면 이유가 더 분명합니다. 표는 `같은 행/열 기준으로 비교되는 정보`를 보존하고, FAQ는 `질문과 답의 짝`을 보존하며, schema는 `이 페이지가 어떤 유형의 정보인지`를 보조 신호로 알려줍니다.

## 표가 중요한 진짜 이유

표는 예쁘게 보이기 위한 편집 요소가 아닙니다. AI 답변은 여러 출처에서 기준을 뽑아 비교하고 요약하는 경우가 많습니다. 이때 긴 문단은 기준과 값을 분리하기 어렵지만, 표는 `항목`, `기준`, `값`, `주의점`을 같은 구조로 묶어 줍니다.

예를 들어 “GEO 도구를 고르는 기준”을 문단으로만 쓰면 가격, 지원 모델, 질문셋 관리, 화면 인용 추적, 리포트 신뢰도가 섞입니다. 하지만 표로 만들면 AI와 독자 모두 어떤 기준으로 비교해야 하는지 바로 알 수 있습니다.

| 문단만 있을 때 | 표로 정리했을 때 | GEO 효과 |
|---|---|---|
| 기준과 예시가 섞인다 | 기준/설명/확인 방법이 분리된다 | AI가 비교 질문에 답하기 쉽다 |
| 중요한 수치가 문장 속에 묻힌다 | 수치와 상태가 한 칸에 고정된다 | 답변 근거로 재사용하기 쉽다 |
| 여러 제품/URL 비교가 흐려진다 | 같은 열 기준으로 비교된다 | co-mention/추천 이유 분석이 쉬워진다 |
| 리라이트 후 무엇이 바뀌었는지 모호하다 | 수정 전/후 항목이 남는다 | 재측정 리포트로 연결하기 쉽다 |

## schema는 본문을 대체하지 않는다

Schema는 AI나 검색엔진에게 “이 페이지가 어떤 종류의 정보인지”를 알려주는 보조 신호입니다. 하지만 schema에만 정보를 넣고 본문에는 없는 상태라면 좋은 구조가 아닙니다. 독자 화면, HTML 본문, schema가 같은 사실을 말해야 합니다.

| 페이지 유형 | 본문에 먼저 있어야 할 것 | 맞는 schema 후보 | 주의할 점 |
|---|---|---|---|
| 용어 설명 | 정의, 예시, 관련 용어 | Article, FAQPage, DefinedTerm | 본문 없는 정의를 schema에만 넣지 않음 |
| 제품/도구 | 기능, 대상, 가격/플랜, 비교 기준 | Product, SoftwareApplication, FAQPage | 오래된 가격/기능 설명이 남지 않게 함 |
| How-to 가이드 | 단계, 조건, 완료 기준 | HowTo, Article, FAQPage | 실제 단계가 본문에 보여야 함 |
| 뉴스룸/보도자료 | 날짜, 조직명, 핵심 팩트, 관련 페이지 | NewsArticle, Organization | 조직명/제품명 표기가 흔들리지 않아야 함 |

## 구조화 도구를 고를 때 먼저 볼 것

| 도구 | 적합한 상황 | 조심할 점 |
|---|---|---|
| FAQ | 반복 질문에 짧게 답할 때 | 본문에 없는 내용을 억지로 넣지 않습니다 |
| 표 | 여러 기준을 비교할 때 | 기준과 항목이 명확해야 합니다 |
| 리스트 | 순서나 체크 항목을 보여줄 때 | 너무 긴 문단을 쪼개는 용도로만 쓰지 않습니다 |
| schema | 검색엔진이 구조를 이해해야 할 때 | 화면 본문과 schema 내용이 달라지면 안 됩니다 |
| 내부 링크 | 관련 질문을 이어 읽게 할 때 | 맥락 없는 링크 나열을 피합니다 |

## 실행 순서

1. 반복 질문과 비교 항목을 분리합니다.
2. 짧은 답변은 FAQ로, 여러 기준 비교는 표로 보냅니다.
3. 구조화 데이터가 필요한 페이지 유형을 표시합니다.
4. schema가 본문 내용과 충돌하지 않는지 확인합니다.
5. 적용 후 크롤러와 검색 노출 변화를 다시 봅니다.

## 예시로 이해하기

`GEO 도구를 고르는 기준`이라는 글을 만든다고 가정해 보겠습니다. 이 주제는 하나의 긴 설명보다 구조화된 정보 단위로 나눌 때 더 강해집니다.

| 정보 단위 | 넣을 내용 | 이유 |
|---|---|---|
| 첫 답변 | GEO 도구를 볼 때는 질문셋 관리, 답변 근거 추적, 화면 인용 확인, 리포트 재현성을 함께 봐야 한다 | 페이지의 결론을 먼저 고정합니다 |
| 표 | 도구별 지원 범위, 측정 지표, 리포트 방식, 검증 방법 | 비교 질문에서 AI가 기준을 재사용하기 쉽습니다 |
| FAQ | “GEO 도구는 SEO 도구와 무엇이 다른가?” 같은 반복 질문 | 질문과 답의 짝을 명확히 남깁니다 |
| schema | Article 또는 FAQPage, 필요하면 SoftwareApplication | 본문에 있는 정보를 구조화 신호로 보강합니다 |
| 내부 링크 | 측정 지표, 답변 근거/화면 인용, 테크니컬 점검 페이지 | 다음 질문으로 이어지는 문맥을 만듭니다 |

이렇게 나누면 FAQ, 표, schema가 서로 대체재가 아니라는 점이 분명해집니다. 표는 비교 기준을 고정하고, FAQ는 짧은 질문에 답하며, schema는 본문에 이미 있는 정보를 검색엔진과 AI가 더 잘 이해하도록 돕습니다. 기본 개념은 [schema.org](https://schema.org/)와 Google의 [구조화 데이터 소개](https://developers.google.com/search/docs/appearance/structured-data/intro-structured-data)를 참고합니다.

## 실습 워크시트

| 입력 항목 | 작성 기준 |
|---|---|
| 본문 질문 | FAQ로 바꿀 질문 |
| 짧은 답변 | 2~3문장 답 |
| 표 기준 | 비교할 축 |
| schema 후보 | FAQPage/HowTo/Article 등 |
| 주의점 | 과장/중복/빈약한 답변 제거 |

## 정리 양식

```text
FAQ 질문 5개 / 각 답변 2문장 / 비교표 열 3개 / 적용할 schema / 검수 체크
```

## 작성 예시

이 예시는 개념을 실제 운영 언어로 바꿔 보는 용도입니다. 그대로 베끼기보다 자기 브랜드의 질문, 페이지, 출처 후보로 바꿔 적습니다.

| 입력 항목 | 작성 예시 |
|---|---|
| FAQ 질문 | GEO 콘텐츠에 FAQ는 반드시 필요한가? |
| 짧은 답변 | 반드시 필요한 것은 아니지만 질문 단위를 명확히 만들 때 유리합니다. |
| 표 기준 | SEO 글 / GEO 글 / AI citation 대응 글 |
| schema 후보 | FAQPage와 Article |
| 주의점 | 같은 질문을 반복하거나 답변을 너무 짧게 쓰지 않는다 |

## 완료 기준

- “FAQ/표/schema는 언제 쓰는가” 기준으로 수정할 문단과 구조가 보입니다.
- 답변 우선 구조, FAQ, 표, schema의 역할이 분리됩니다.
- 리라이트 후 다시 확인할 질문셋이 남습니다.

## 흔한 질문

**Q. 이 내용을 이 책에서 바로 실습으로 써도 되나요?**

가능합니다. 다만 먼저 `FAQ/표/schema는 언제 쓰는가`에서 어떤 판단 기준을 얻는지 설명한 뒤 실습으로 넘어가는 편이 좋습니다. 표를 채우는 것보다 왜 이 기준을 쓰는지 이해시키는 것이 먼저입니다.

**Q. 모든 항목을 한 번에 실행해야 하나요?**

아닙니다. FAQ, 표, schema는 필요한 곳에만 써야 합니다. 질문이 짧은 답을 요구할 때 FAQ를 쓰고, 비교 기준이 많을 때 표를 씁니다.
