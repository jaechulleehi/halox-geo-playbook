## 기존 글을 GEO 콘텐츠로 리라이트하는 체크리스트

GEO 리라이트는 새 글을 많이 쓰기 전에 기존 자산을 답변형 구조로 바꾸는 작업입니다. 첫 문단, 질문형 H2, 표, FAQ, 출처, 업데이트 맥락을 순서대로 고칩니다.

많은 팀이 새 글 발행부터 시작하지만 이미 가진 글의 구조가 약해서 AI 답변에 쓰이지 못하는 경우가 많습니다.

## 리라이트는 HTML 구조까지 바꾸는 일

GEO 리라이트를 문장 수정으로만 보면 4장의 핵심을 놓칩니다. 실제로는 AI가 페이지를 읽을 때 남는 텍스트 구조를 바꾸는 일입니다. 제목, H2, 첫 문단, 표, FAQ, 내부 링크, schema, 이미지 alt, 업데이트 날짜가 서로 같은 주제를 가리키게 만들어야 합니다.

| 리라이트 대상 | 문장 수정만 한 경우 | GEO 리라이트 기준 |
|---|---|---|
| 제목/H2 | 키워드만 추가 | 질문 의도와 답변 범위를 드러냄 |
| 첫 문단 | 자연스럽게 다듬음 | 정의/결론/적용 범위/다음 행동을 먼저 제시 |
| 본문 | 문단을 늘림 | 기준, 예외, 비교, 근거를 분리 |
| 표/FAQ | 보기 좋게 추가 | AI가 재사용할 질문/비교 단위로 정리 |
| schema | 플러그인으로 자동 생성 | 본문 내용과 같은 사실을 구조화 |
| 내부 링크 | 관련 글을 하단에 나열 | 문맥 안에서 다음 질문으로 연결 |

## 리라이트 전에 먼저 볼 것

| 점검 항목 | 확인 질문 | 다음 액션 |
|---|---|---|
| 질문 | 이 페이지가 답하는 핵심 질문은 무엇인가? | 질문을 한 문장으로 다시 쓴다 |
| 자산 | 이미 답할 콘텐츠나 페이지가 있는가? | 신규/리라이트/보강으로 나눈다 |
| 출처 | AI가 참고할 근거가 충분한가? | 내부 근거와 외부 근거를 분리한다 |
| 측정 | 어떤 지표로 변화를 볼 것인가? | mention, 답변 근거(source), 화면 인용(citation), co-mention 중 핵심 지표를 고른다 |

## 실행 순서

1. 기존 글의 핵심 질문을 다시 씁니다.
2. 첫 답변, 근거, 비교, FAQ, 출처 연결을 순서대로 점검합니다.
3. 빠진 문단과 과한 문단을 표시합니다.
4. 수정할 위치와 담당자를 정합니다.
5. 수정 후 재측정할 질문셋을 붙입니다.

## 사례로 이해하기

검색용 글을 GEO 글로 바꿀 때는 키워드만 추가하면 안 됩니다. 첫 답변, 비교 기준, 인용 가능한 문장, 출처 링크, FAQ, 다음 액션이 함께 들어가야 AI 답변 재료가 됩니다.

## 리라이트가 단순 수정이 아닌 이유

GEO 리라이트는 문장을 예쁘게 고치는 일이 아닙니다. 기존 글이 어떤 질문에 답하지 못하는지, 어떤 기준을 빠뜨렸는지, 어떤 출처와 연결되지 않는지를 다시 설계하는 작업입니다.

따라서 리라이트 전에 먼저 질문셋과 콘텐츠 갭을 봐야 합니다. 질문이 없으면 수정 기준이 감각에 머물고, 갭이 없으면 불필요한 문장만 늘어납니다. 좋은 리라이트는 첫 답변, 비교 기준, FAQ, 표, source 연결 중 무엇을 고칠지 분명합니다.

## 실습 워크시트

| 입력 항목 | 작성 기준 |
|---|---|
| 기존 문단 | 바꾸기 전 문장 |
| 문제 | 늦은 답변/근거 부족/구조 없음 |
| 리라이트 문장 | Answer-first로 수정 |
| 추가 요소 | 표/FAQ/출처/예시 |
| 검수 | 인용 가능한가 |

## 정리 양식

```text
기존 URL / 목표 질문 / 바꾸기 전 첫 문단 / 바꾼 첫 문단 / 추가한 표/FAQ / 연결 링크 / 재측정 질문
```

## 작성 예시

이 예시는 개념을 실제 운영 언어로 바꿔 보는 용도입니다. 그대로 베끼기보다 자기 브랜드의 질문, 페이지, 출처 후보로 바꿔 적습니다.

| 입력 항목 | 작성 예시 |
|---|---|
| 기존 문단 | GEO 최적화를 소개하는 글입니다. |
| 문제 | 질문에 대한 답이 늦고 인용 가능한 기준이 없다 |
| 리라이트 문장 | GEO는 AI 답변 안에서 브랜드가 발견되고 비교되고 인용되도록 콘텐츠와 출처를 설계하는 작업입니다. |
| 추가 요소 | 답변 근거(source)/화면 인용(citation) 차이 표, FAQ 3개, HaloX AVI 점수 링크 |
| 검수 | 첫 150자 안에 결론과 핵심 키워드가 들어갔다 |

## 완료 기준

- “기존 글을 GEO 콘텐츠로 리라이트하는 체크리스트” 기준으로 수정할 문단과 구조가 보입니다.
- 답변 우선 구조, FAQ, 표, schema의 역할이 분리됩니다.
- 리라이트 후 다시 확인할 질문셋이 남습니다.

## 참고 링크 패키지

리라이트의 목표가 단순한 문장 개선이 아니라 AI 답변의 근거가 되는 것이라면 HaloX의 [AI에게 인용되는 콘텐츠 가이드](https://haloxlabs.ai/ko/blog/how-to-get-cited-by-ai)를 함께 봅니다. 이 페이지의 체크리스트는 고칠 항목을 정하고, HaloX 글은 인용 가능성을 높이는 콘텐츠 조건을 설명합니다.

리라이트 후 구조화 데이터까지 손볼 때는 Google의 [Article structured data](https://developers.google.com/search/docs/appearance/structured-data/article)와 [구조화 데이터 소개](https://developers.google.com/search/docs/appearance/structured-data/intro-structured-data)를 참고합니다. 단, schema는 본문에 실제로 있는 정보를 명확히 표현하는 보조 장치로만 써야 합니다.

## 체크리스트

- 원문에서 첫 답변이 늦게 나오지 않는가?
- 삭제할 문장과 보강할 문장이 구분됐는가?
- 답변 근거(source)/화면 인용(citation) 후보가 본문 안에 자연스럽게 붙는가?
- 수정 담당자와 기한이 정해졌는가?
- 수정 후 같은 질문으로 다시 확인할 계획이 있는가?

## 흔한 질문

**Q. 이 내용을 이 책에서 바로 실습으로 써도 되나요?**

가능합니다. 다만 먼저 `기존 글을 GEO 콘텐츠로 리라이트하는 체크리스트`에서 어떤 판단 기준을 얻는지 설명한 뒤 실습으로 넘어가는 편이 좋습니다. 표를 채우는 것보다 왜 이 기준을 쓰는지 이해시키는 것이 먼저입니다.

**Q. 모든 항목을 한 번에 실행해야 하나요?**

아닙니다. 먼저 조회수나 전환 가치가 큰 페이지를 고르고, 첫 답변과 근거 구조부터 고치는 편이 좋습니다.

## 리라이트 전후 비교표

기존 글을 GEO 콘텐츠로 바꿀 때는 문장을 조금 다듬는 수준에서 멈추면 안 됩니다. 아래처럼 전후를 나눠 보면 어떤 부분을 실제로 고쳤는지 보입니다.

| 항목 | 리라이트 전 | 리라이트 후 |
|---|---|---|
| 첫 문단 | 배경 설명이 길고 결론이 늦게 나옴 | 첫 3~5문장 안에 정의/결론/대상 독자 제시 |
| 제목 구조 | 키워드 중심 제목 반복 | 독자 질문과 판단 기준 중심 H2 |
| 비교 정보 | 문단 안에 흩어져 있음 | 표로 기준/값/주의점 분리 |
| FAQ | 없음 또는 형식적 질문 | 실제 후속 질문 3~5개와 짧은 답변 |
| 출처 링크 | 내부 링크만 있음 | 공식 자료/외부 출처/관련 장 링크 분리 |
| 기술 전달 | 개발팀에 요청할 내용이 없음 | schema/내부 링크/렌더링 점검 항목을 06장으로 넘김 |

완료 기준은 “좋아 보이는 글”이 아니라, 02장에서 측정한 질문에 다시 넣었을 때 답변 근거와 화면 인용 후보가 더 분명해지는 것입니다.
