## AI가 읽기 좋은 콘텐츠 구조: HTML에서 답변 재료까지

![AI가 읽기 좋은 콘텐츠 구조](../assets/images/chapter-heroes/halox-geo-ai-readable-content-hero-1600x900.png)

이 장은 `AI가 읽기 좋은 콘텐츠 구조`를 문체 문제가 아니라 **기계가 읽는 정보 구조**의 문제로 다룹니다. GEO에서 좋은 글은 사람이 읽기 좋은 글이면서, AI가 URL, 제목, 설명, 본문, 표, FAQ, 링크, schema를 같은 의미로 해석할 수 있는 글입니다.

스터디에서 반복해서 다룬 핵심은 간단합니다. AI는 사람처럼 화면을 감상하는 것이 아니라, 먼저 URL과 제목/설명 같은 단서를 보고 페이지에 접근한 뒤 HTML을 가져오고, 필요하면 렌더링한 DOM을 읽고, 그 내용을 Markdown에 가까운 텍스트 구조로 바꿔 답변 재료를 고릅니다. 이 과정에서 첫 답변, H2, 표, FAQ, 내부 링크, schema가 서로 같은 말을 해야 답변 근거(source)와 화면 인용(citation) 가능성이 높아집니다.

이 책에서는 3주차 리라이트 실습과 직접 연결됩니다. 각 세부 페이지는 원문을 어떻게 고치고, 어떤 표/FAQ/schema/출처 구조를 붙일지 판단하는 기준으로 사용합니다.

## 콘텐츠 리라이트 패키지

이 장의 세부 페이지는 `첫 답변 → 구조화 요소 → 리라이트 체크리스트` 순서로 이어집니다. 기존 글을 가져와 첫 문단, H2, 표, FAQ, schema, 출처 링크를 바꾸는 실습에 바로 쓸 수 있습니다.

## AI가 페이지를 읽는 과정을 먼저 이해하기

AI가 웹 콘텐츠를 답변 재료로 쓸 때의 흐름은 대략 다음 순서로 이해하면 쉽습니다. 실제 플랫폼마다 세부 구현은 다르지만, 콘텐츠를 설계할 때는 이 관점이 유용합니다.

| 단계 | AI가 보는 것 | 콘텐츠에서 신경 쓸 것 |
|---|---|---|
| URL 발견 | 검색 결과, 링크, sitemap, 내부 링크, 외부 인용 | URL이 주제를 설명하고 핵심 페이지가 발견 가능한가 |
| 제목/설명 판단 | title, meta description, H1/H2, 주변 앵커 텍스트 | 어떤 질문에 답하는 페이지인지 초반에 드러나는가 |
| HTML 수집 | 서버 응답 HTML, 렌더링 후 DOM, 본문 텍스트 | 중요한 답변 재료가 HTML 안에 실제 텍스트로 있는가 |
| Markdown식 구조화 | 제목, 문단, 표, 리스트, FAQ, 링크를 텍스트 구조로 변환 | 표/리스트/FAQ가 의미 단위로 분리되어 있는가 |
| 답변 합성 | 여러 출처의 정의, 비교 기준, 예외, 근거를 조합 | 첫 답변, 기준표, 출처 링크, 업데이트 맥락이 일관적인가 |

그래서 4장은 “글을 예쁘게 쓰는 법”이 아니라 “AI가 읽고 재사용할 수 있는 정보 단위를 만드는 법”입니다. 6장은 이 구조가 실제 HTML/렌더링/schema/크롤러 접근성에서도 유지되는지 확인합니다.

## 실행 체크리스트로 바꾸는 법

04장은 읽고 끝나는 장이 아니라 기존 콘텐츠를 고치는 기준표로 써야 합니다. 03장에서 만든 질문셋을 가져와 아래 순서로 확인합니다.

| 점검 순서 | 확인 질문 | 통과 기준 | 다음 액션 |
|---|---|---|---|
| 첫 답변 | 첫 3~5문장 안에 결론이 있는가 | 정의/대상/조건/주의점이 초반에 보인다 | 첫 문단 재작성 |
| H2 구조 | 질문별로 제목이 나뉘는가 | H2만 훑어도 답변 흐름이 보인다 | 제목을 질문형/판단형으로 수정 |
| 표/리스트 | 비교 기준이 구조화되어 있는가 | 기준/값/주의점이 분리되어 있다 | 표 또는 체크리스트 추가 |
| FAQ | 독자가 실제로 묻는 후속 질문이 있는가 | 짧은 답과 다음 링크가 있다 | FAQ 3~5개 작성 |
| schema | 본문 의미를 보조하는가 | 본문/title/canonical/schema가 같은 의미를 말한다 | 06-05로 넘겨 기술 점검 |
| 내부 링크 | 다음 질문으로 이어지는가 | 관련 장/하위 페이지 링크가 자연스럽다 | 콘텐츠 허브 링크 정리 |

이 표를 채우면 04장은 콘텐츠팀의 리라이트 작업표가 되고, 06장은 개발팀의 구현 점검표가 됩니다.

## HaloX로 이어지는 지점

AI가 읽기 좋은 글 구조는 HaloX의 [AI 선호 콘텐츠 구조](https://haloxlabs.ai/ko/blog/ai-preferred-content-structure)와 함께 보면 좋습니다. 4장은 리라이트 실습을, HaloX 글은 외부 독자에게 설명하기 쉬운 원칙을 제공합니다. 콘텐츠 구조를 고칠 때는 Google의 [구조화 데이터 소개](https://developers.google.com/search/docs/appearance/structured-data/intro-structured-data)와 [유용한 콘텐츠 만들기](https://developers.google.com/search/docs/fundamentals/creating-helpful-content)를 함께 봅니다. schema는 본문을 대체하는 장치가 아니라, 본문 의미를 더 명확히 전달하는 보조 신호입니다.
