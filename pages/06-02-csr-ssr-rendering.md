## CSR/SSR 렌더링은 GEO에서 왜 중요한가

CSR/SSR 점검은 핵심 본문이 초기 HTML에서 보이는지, crawler가 안정적으로 읽을 수 있는지 확인하는 작업입니다. AI 검색에서는 렌더링 전후 차이가 source 채택에 영향을 줄 수 있습니다.

검색과 AI crawler가 모든 JavaScript를 완벽하게 실행한다고 가정하면 위험합니다. 중요한 본문이 늦게 나타나면 일부 환경에서 빠질 수 있습니다.

## 실무에서 먼저 볼 것

| 점검 항목 | 확인 질문 | 다음 액션 |
|---|---|---|
| 질문 | 이 페이지가 답하는 핵심 질문은 무엇인가? | 질문을 한 문장으로 다시 쓴다 |
| 자산 | 이미 답할 콘텐츠나 페이지가 있는가? | 신규/리라이트/보강으로 나눈다 |
| 출처 | AI가 참고할 근거가 충분한가? | 내부 근거와 외부 근거를 분리한다 |
| 측정 | 어떤 지표로 변화를 볼 것인가? | mention/source/citation/co-mention 중 핵심 지표를 고른다 |

## 실행 순서

1. 대표 질문을 하나 고릅니다.
2. 그 질문에 답하기 위해 필요한 정보 단위를 적습니다.
3. 현재 보유한 콘텐츠, 제품 페이지, 외부 출처를 연결합니다.
4. 부족한 부분을 신규 작성, 리라이트, 오프사이트 보강, 기술 점검으로 분류합니다.
5. 같은 질문셋으로 다시 측정할 시점을 정합니다.

## 사례로 이해하기

사용자에게는 보이지만 초기 HTML에 핵심 문장이 없으면 AI crawler가 내용을 놓칠 수 있습니다. CSR/SSR 차이를 기술 문제가 아니라 답변 재료 노출 문제로 봐야 합니다.

## 실습 워크시트

| 입력 항목 | 작성 기준 |
|---|---|
| 페이지 | 점검 URL |
| 초기 HTML 핵심 내용 | 있음/없음 |
| 렌더링 방식 | CSR/SSR/SSG/혼합 |
| 빠진 정보 | 제목/본문/FAQ/가격/비교표 |
| 수정안 | SSR/정적화/본문 위치 조정 |

## 제출 템플릿

```text
URL / View-source 확인 결과 / 렌더링 방식 / 누락 콘텐츠 / 개발 요청 문장 / 재검수 기준
```

## 작성 예시

아래 예시는 `AcmeGEO`라는 가상의 B2B SaaS 브랜드를 기준으로 작성한 샘플입니다. 실제 강의에서는 브랜드명, 질문, 출처 후보, 담당자를 자기 상황에 맞게 바꿉니다.

| 입력 항목 | 작성 예시 |
|---|---|
| 페이지 | /ko/glossary/geo |
| 초기 HTML 핵심 내용 | 용어 제목은 있으나 설명 본문이 늦게 로드됨 |
| 렌더링 방식 | CSR 중심 |
| 빠진 정보 | 정의 문단과 관련 FAQ |
| 수정안 | 정의 문단과 FAQ를 서버 렌더링 또는 정적 생성으로 노출한다 |

## 완료 기준

- 작성한 표만 봐도 다음 액션이 보입니다.
- 질문, 콘텐츠, 출처, 측정 기준이 서로 연결되어 있습니다.
- 강의 후 제출물로 쓰거나 이북 워크시트로 옮겨도 설명이 부족하지 않습니다.

## 흔한 질문

**Q. 이 페이지의 내용을 바로 강의자료로 써도 되나요?**

가능합니다. 다만 강의에서는 설명보다 실습 산출물을 먼저 정해야 합니다.

**Q. 모든 항목을 한 번에 실행해야 하나요?**

아닙니다. 현재 가장 중요한 질문군 하나를 고르고, 그 질문에서 보이는 콘텐츠/출처/기술 갭부터 처리합니다.
