## 사이트링크와 리치 리절트 차이: 검색결과 섹션 링크는 어떻게 만들어지나
![사이트링크와 리치 리절트 차이: 검색결과 섹션 링크는 어떻게 만들어지나 대표 이미지](../assets/images/page-heroes/halox-geo-06-09-sitelinks-rich-results-section-links-hero.png)

Google 검색결과에서 한 결과 아래에 `Schema 타입별 점검표`, `FAQPage schema 점검`, `Google 공식 도구 검증`처럼 작은 버튼형 링크가 붙는 경우가 있습니다. 이 노출은 보통 FAQ/HowTo schema 같은 리치 리절트라기보다, Google이 페이지의 목차, 내부 앵커, H2/H3 헤딩 구조를 해석해 자동으로 보여주는 사이트링크 또는 페이지 섹션 링크에 가깝습니다.

GEO 관점에서는 이 차이가 중요합니다. 리치 리절트는 구조화 데이터와 Google이 지원하는 검색결과 유형의 문제이고, 섹션 링크는 페이지 내부 구조와 헤딩 설계의 문제입니다. 둘을 구분해야 콘텐츠팀은 헤딩/목차를 고치고, 개발팀은 HTML 앵커/schema/렌더링을 제대로 점검할 수 있습니다.

## 한 문장으로 구분하면

리치 리절트는 `구조화 데이터로 검색결과 표현을 풍부하게 만드는 장치`이고, 사이트링크/페이지 섹션 링크는 `Google이 사이트 구조와 페이지 내부 링크를 읽고 사용자가 바로 이동할 만한 지점을 자동으로 보여주는 링크`입니다.

| 구분 | 사이트링크/페이지 섹션 링크 | 리치 리절트 |
|---|---|---|
| 핵심 성격 | 검색결과 안의 추가 이동 링크 | 일반 검색결과보다 풍부한 표현 |
| 주요 근거 | 사이트 구조, 내부 링크, 목차, 앵커, 헤딩, URL 구조 | schema.org 기반 구조화 데이터, Google 지원 유형 |
| 예시 | `Schema 타입별 점검표`, `FAQPage schema 점검`, `Google 공식 도구 검증` 같은 섹션 링크 | FAQ, 상품 가격/평점, 이벤트, 레시피, breadcrumb 등 |
| 제어 가능성 | 직접 지정 불가, 구조를 통해 확률을 높임 | schema를 구현하고 검증할 수 있지만 노출 보장은 아님 |
| 확인 도구 | 실제 검색결과, Search Console 성과, HTML/DOM 확인 | Rich Results Test, Schema Markup Validator |
| GEO 의미 | AI와 검색엔진이 페이지 안의 답변 단위를 이해하기 쉬워짐 | 페이지의 엔터티/질문/상품/조직 정보를 기계가 읽기 쉬워짐 |

따라서 검색결과에 작은 링크가 붙었다고 해서 바로 “리치 리절트가 떴다”고 보면 안 됩니다. 그 링크가 `FAQ 결과`, `상품 정보`, `평점`, `이벤트`처럼 구조화 데이터 기반 표현인지, 아니면 같은 페이지 안의 특정 섹션으로 이동하는 링크인지 먼저 봐야 합니다.

## WikiDocs 검색결과에서 보인 것은 무엇인가

GEO WikiDocs의 테크니컬 페이지 검색결과에서 `Schema 타입별 점검표`, `FAQPage schema 점검`, `Google 공식 도구 검증` 같은 항목이 노출된 경우는 페이지 섹션 링크로 보는 편이 자연스럽습니다.

이유는 간단합니다.

1. 해당 문구가 본문 안의 H2/H3 섹션명과 직접 대응합니다.
2. WikiDocs의 `[TOC]`가 HTML에서 내부 앵커 링크로 렌더링됩니다.
3. 검색결과 링크를 누르면 별도 FAQ 카드가 아니라 같은 문서의 특정 위치로 이동하는 성격입니다.
4. Rich Results Test로 확인하는 FAQ/Product/Event 같은 구조화 데이터 노출과 다릅니다.

즉, Google이 “이 페이지 안에서 사용자가 바로 갈 만한 섹션”을 찾아 검색결과 아래에 붙인 것입니다. 공식 용어로는 사이트링크에 가까운 자동 링크이고, 실무에서는 페이지 섹션 링크, 점프 링크, 앵커 링크 노출이라고 설명하면 이해하기 쉽습니다.

## HTML 구조는 어떻게 생겼나

WikiDocs에서 `[TOC]`를 넣으면 렌더링된 HTML에는 대체로 목차 영역과 본문 헤딩 앵커가 만들어집니다. 핵심은 목차의 `href="#..."`와 본문 헤딩의 `id="..."`가 연결되는 구조입니다.

```html
<div class="toc">
  <ul>
    <li><a href="#schema-타입별-점검표">Schema 타입별 점검표</a></li>
    <li><a href="#faqpage-schema-점검">FAQPage schema 점검</a></li>
    <li><a href="#google-공식-도구-검증">Google 공식 도구 검증</a></li>
  </ul>
</div>

<h3 id="schema-타입별-점검표">Schema 타입별 점검표</h3>
<p>Organization, Person/ProfilePage, FAQPage, Product schema를 페이지 유형별로 확인합니다.</p>

<h3 id="faqpage-schema-점검">FAQPage schema 점검</h3>
<p>본문에 실제 질문과 답변이 있고 JSON-LD와 충돌하지 않는지 확인합니다.</p>

<h3 id="google-공식-도구-검증">Google 공식 도구 검증</h3>
<p>Rich Results Test, Schema Markup Validator, Search Console을 함께 확인합니다.</p>
```

검색엔진 입장에서는 이 구조가 매우 읽기 쉽습니다. 페이지 초반에 목차가 있고, 목차 링크가 본문 헤딩과 연결되며, 헤딩명이 사용자가 검색할 만한 문구라면 “이 페이지에는 여러 답변 단위가 있다”고 판단하기 좋아집니다.

## 왜 `[TOC]`가 도움이 되는가

`[TOC]`는 단순한 독자 편의 기능이 아닙니다. WikiDocs처럼 목차가 실제 내부 링크로 렌더링되는 환경에서는 검색엔진과 AI가 페이지 구조를 파악하는 데도 도움이 됩니다.

좋은 `[TOC]` 구조는 다음 신호를 줍니다.

| 신호 | 의미 |
|---|---|
| 페이지 초반 목차 | 이 문서가 어떤 하위 질문을 다루는지 먼저 보여줌 |
| 내부 앵커 링크 | 각 섹션으로 바로 이동할 수 있는 구조 제공 |
| H2/H3와 목차 텍스트 일치 | 목차와 본문이 같은 주제를 말한다는 신호 |
| 검색 의도형 헤딩 | 사용자가 검색할 문구와 섹션명이 가까움 |
| 중복 없는 섹션 구분 | Google이 어떤 링크를 보여줄지 판단하기 쉬움 |

GEO에서는 이 구조가 AI 답변 근거(source)와 화면 인용(citation)에도 영향을 줄 수 있습니다. AI가 페이지 전체를 하나의 긴 글로만 보는 것이 아니라, `정의`, `비교`, `체크리스트`, `주의사항`, `실습` 같은 답변 단위로 나눠 이해하기 쉬워지기 때문입니다.

## 어떤 기준으로 섹션 링크가 노출될까

Google이 정확한 노출 공식을 공개하는 것은 아니지만, 사이트링크 관련 공식 문서는 사이트의 링크 구조를 분석해 사용자에게 유용한 지름길을 자동으로 찾는다고 설명합니다. 실무에서는 아래 조건이 중요합니다.

| 기준 | 좋은 상태 | 나쁜 상태 |
|---|---|---|
| 목차 위치 | 첫 답변/도입부 뒤에 `[TOC]`가 있음 | 목차가 없거나 너무 아래에 있음 |
| 내부 앵커 | 목차 링크가 실제 본문 헤딩 id로 연결됨 | 클릭해도 이동하지 않거나 JS로만 처리됨 |
| 헤딩 문구 | `GEO 뜻`, `리치 리절트 차이`, `Schema 점검표`처럼 구체적 | `개요`, `참고`, `마무리`처럼 모호함 |
| 섹션 수 | 핵심 섹션 3~8개가 선명함 | 너무 많거나 너무 적고 위계가 흐림 |
| 본문 일치 | 헤딩 아래 본문이 해당 질문에 바로 답함 | 헤딩은 좋은데 내용은 다른 이야기 |
| 검색 의도 | 사용자가 실제로 찾는 문제와 연결됨 | 내부 운영자만 이해하는 표현 |
| 사이트 신뢰 | 색인 가능, canonical 안정, 내부 링크 정상 | noindex, 중복 URL, 느린 렌더링, 깨진 링크 |

가장 중요한 것은 “Google에 이 링크를 띄워 달라”고 요청하는 기능이 아니라는 점입니다. 우리는 구조를 정리해 확률을 높일 수 있을 뿐이고, 실제 노출 여부와 문구 선택은 Google 시스템이 검색어/사용자 의도/페이지 품질을 보고 자동으로 결정합니다.

## GEO 콘텐츠에서는 어떤 헤딩이 유리한가

섹션 링크로 잡히기 좋은 헤딩은 짧고, 구체적이고, 검색 의도가 보이는 문구입니다. 특히 GEO 위키북스에서는 독자가 실제로 묻는 질문과 HaloX가 측정하는 지표가 헤딩에 자연스럽게 드러나야 합니다.

| 약한 헤딩 | 좋은 헤딩 |
|---|---|
| 개요 | GEO 뜻: 생성형 엔진 최적화란 무엇인가 |
| 차이점 | GEO/SEO/AEO/AIO/LLMO 차이 |
| 지표 | mention/source/citation 지표 해석법 |
| 도구 | GEO 도구와 SEO 도구 비교 |
| 구조 | Answer-first 콘텐츠 구조 |
| 스키마 | Schema 타입별 점검표 |
| 주의사항 | 리치 리절트와 사이트링크를 혼동하면 생기는 문제 |
| 마무리 | Search Console로 재측정할 항목 |

헤딩은 검색어를 억지로 나열하는 공간이 아닙니다. 하지만 검색결과에 잘려서 보이더라도 의미가 남아야 합니다. `확인`, `주의사항`, `정리`보다 `Schema 타입별 점검표`, `GEO 리포트 확인`, `FAQPage schema 점검표`가 더 좋습니다.

## 리치 리절트가 필요한 경우와 아닌 경우

모든 검색결과 개선을 schema로 해결하려고 하면 작업이 꼬입니다. 먼저 원하는 노출이 리치 리절트인지, 섹션 링크인지 구분해야 합니다.

| 원하는 결과 | 먼저 볼 것 | 이유 |
|---|---|---|
| 검색결과 아래에 본문 섹션 바로가기 노출 | `[TOC]`, H2/H3, 내부 앵커, 사이트 구조 | 같은 페이지 안의 특정 답변으로 이동하는 문제 |
| FAQ 질문/답변 형태의 특수 노출 | FAQ 본문, FAQPage schema, Google 지원 정책 | 구조화 데이터와 본문 일치가 필요 |
| 상품 가격/재고/리뷰 노출 | Product/Offer/Review schema, merchant feed | 상품 데이터 구조화가 필요 |
| 조직명/로고/프로필 일관성 | Organization/ProfilePage schema, sameAs, 외부 프로필 | 엔터티 신호 정리가 필요 |
| 검색결과 제목/설명 개선 | title, meta description, 본문 첫 문단 | snippet/title link 생성 신호 문제 |
| breadcrumb 노출 | BreadcrumbList schema, URL/카테고리 구조 | 페이지 위치를 구조화하는 문제 |

리치 리절트는 “본문에 있는 정보를 구조화 데이터로 표현하는 작업”입니다. 반면 섹션 링크는 “본문의 답변 단위가 명확하고 이동 가능한 구조로 나뉘어 있는가”의 문제입니다. 둘은 함께 갈 수 있지만 같은 작업은 아닙니다.

## 섹션 링크를 GEO 목차 설계로 활용하기

사이트링크와 섹션 링크는 직접 제어할 수 없지만, 좋은 H2 구조와 TOC는 검색결과와 AI 답변 모두에서 페이지의 하위 질문을 이해시키는 데 도움이 됩니다.

| H2 유형 | 약한 예 | 좋은 예 |
|---|---|---|
| 정의 | 개요 | GEO 도구는 SEO 도구와 무엇이 다른가 |
| 비교 | 기능 | mention/source/citation을 따로 보여주는가 |
| 실행 | 방법 | 첫 30일 동안 어떤 질문셋을 측정할까 |
| 검증 | 참고 | 리포트가 믿을 만한지 어떤 source를 볼까 |
| 리스크 | 주의사항 | 과장된 GEO 제안서는 어떻게 걸러낼까 |

H2가 독자의 하위 질문처럼 쓰이면 검색 섹션 링크 후보가 좋아지고, AI가 문서를 Markdown처럼 읽을 때도 답변 단위를 분리하기 쉽습니다.

## 실무 점검 순서

사이트링크와 리치 리절트를 함께 점검할 때는 아래 순서가 안전합니다.

1. 실제 검색결과에서 노출된 작은 링크를 클릭해 같은 페이지의 특정 섹션으로 이동하는지 확인합니다.
2. 페이지 소스 또는 렌더링 DOM에서 `[TOC]`가 `<div class="toc">`, `<a href="#...">`, `h2/h3 id="..."` 구조로 나오는지 확인합니다.
3. 해당 섹션명이 본문 H2/H3와 일치하는지 확인합니다.
4. 헤딩명이 검색 의도형인지, 너무 짧거나 모호하지 않은지 점검합니다.
5. Rich Results Test로 해당 URL이 Google 지원 리치 리절트 대상인지 별도로 확인합니다.
6. Schema Markup Validator로 JSON-LD 문법과 타입 오류를 확인합니다.
7. Search Console에서 해당 URL의 검색어/노출/클릭 변화를 봅니다.
8. 수정 후에는 같은 검색어와 같은 URL로 재측정합니다.

이 순서의 핵심은 “검색결과에 보이는 모든 특수한 모양을 schema 문제로 몰지 않는 것”입니다. 섹션 링크는 콘텐츠 구조와 HTML 앵커를, 리치 리절트는 구조화 데이터를 봐야 합니다.

## 개발팀에 넘길 HTML 체크리스트

콘텐츠팀이 헤딩을 잘 써도 렌더링된 HTML이 읽히지 않으면 효과가 약해집니다. 개발팀에는 다음 항목을 URL 단위로 넘기는 편이 좋습니다.

| 점검 항목 | 확인 방법 | 완료 기준 |
|---|---|---|
| 목차 렌더링 | View source/DOM 검사 | 목차가 실제 링크 목록으로 보임 |
| 앵커 연결 | 목차 클릭 테스트 | `href="#id"`가 본문 `id`와 연결됨 |
| 초기 HTML | 소스 보기/렌더링 비교 | 핵심 H2/H3와 첫 답변이 텍스트로 확인됨 |
| 헤딩 위계 | DOM에서 h2/h3 순서 확인 | H1 남발 없이 페이지 제목/H2/H3 위계가 자연스러움 |
| canonical | head 태그 확인 | 대표 URL이 일관됨 |
| robots meta | head 태그 확인 | 핵심 페이지가 index/follow 상태 |
| schema | Rich Results Test/Validator | 본문과 같은 정보만 JSON-LD로 표현됨 |
| 내부 링크 | a 태그 확인 | 관련 페이지가 모호한 “자세히”가 아니라 의미 있는 앵커로 연결됨 |

WikiDocs처럼 플랫폼이 목차와 앵커를 자동 렌더링하는 경우에는 콘텐츠 규칙이 곧 기술 구조가 됩니다. 그래서 본문에 `[TOC]`를 넣고, H2/H3 문구를 정리하는 일이 GEO 기술 점검과 직접 연결됩니다.

## HaloX 리포트에서는 어떻게 봐야 하나

HaloX 리포트에서 특정 URL이 AI 답변 근거(source)로는 잡히지만 화면 인용(citation)이 약하다면, 페이지 안의 답변 단위가 명확한지 봐야 합니다. AI가 페이지를 발견해도 섹션 구조가 흐리면 “어느 부분을 인용해야 하는지”가 약해질 수 있습니다.

| 리포트 신호 | 의심할 구조 문제 | 수정 방향 |
|---|---|---|
| source는 잡히지만 citation이 약함 | 페이지 안의 답변 단위가 흐림 | H2/H3와 첫 문단을 질문형으로 정리 |
| 브랜드 언급은 되지만 근거 URL이 불안정함 | canonical/내부 링크/허브 구조 약함 | 대표 URL과 허브 링크 정리 |
| AI가 오래된 내용을 답함 | 업데이트 날짜/schema/본문 불일치 | 본문과 구조화 데이터 동시 갱신 |
| 검색결과 섹션 링크가 엉뚱함 | 헤딩명이 모호하거나 중복됨 | 섹션명 중복 제거, 검색 의도형 헤딩 사용 |
| Rich Results Test는 통과하지만 답변 품질이 약함 | schema는 있으나 본문 답변이 약함 | Answer-first 본문, FAQ, 표 보강 |

GEO에서 목표는 “검색결과 장식”이 아닙니다. 목표는 사용자가 묻는 질문에 대해 검색엔진과 AI가 같은 URL, 같은 섹션, 같은 설명을 안정적으로 이해하게 만드는 것입니다.

## 흔한 오해

**Q. 검색결과 아래 작은 링크가 보이면 무조건 리치 리절트인가요?**

아닙니다. 같은 페이지의 특정 섹션으로 이동하는 링크라면 리치 리절트보다 사이트링크/페이지 섹션 링크로 보는 편이 맞습니다.

**Q. `[TOC]`만 넣으면 Google 검색결과에 섹션 링크가 반드시 나오나요?**

아닙니다. `[TOC]`는 구조를 명확히 하는 신호입니다. 실제 노출은 검색어, 페이지 품질, 사이트 구조, Google 시스템 판단에 따라 달라집니다.

**Q. 리치 리절트와 섹션 링크를 동시에 노릴 수 있나요?**

가능합니다. 예를 들어 FAQ 본문을 H2/H3로 잘 정리하고, 본문에 실제 FAQ를 넣고, FAQPage schema를 본문과 일치하게 구현하면 섹션 구조와 구조화 데이터 신호를 함께 만들 수 있습니다. 다만 둘의 검증 기준은 다릅니다.

**Q. GEO에서는 schema보다 헤딩이 더 중요한가요?**

둘 중 하나만 고르는 문제가 아닙니다. 헤딩과 목차는 답변 단위를 만들고, schema는 본문 정보를 기계가 읽기 쉽게 라벨링합니다. 좋은 상태는 본문, HTML, 목차, 내부 링크, schema, canonical이 같은 의미를 말하는 구조입니다.
