## GEO 제안서는 어떤 구조로 써야 할까

GEO 제안서는 기능 소개서가 아니라 문제를 증명하고 실행 범위를 합의하는 문서여야 합니다. 좋은 제안서는 고객의 현재 AI 답변 상태, 놓친 질문 시장, source/citation 부족, 콘텐츠/기술/오프사이트 액션, 재측정 기준을 한 흐름으로 보여줍니다.

특히 대행사나 컨설팅 관점에서는 제안서가 곧 신뢰의 첫 리포트입니다. 제안서에서 질문과 지표가 모호하면 실제 운영 리포트도 모호해질 가능성이 높습니다.

## 제안서 기본 목차

| 제안서 섹션 | 들어갈 내용 | 주의할 점 |
|---|---|---|
| 1. 현재 문제 | AI 답변에서 보이지 않거나 잘못 설명되는 질문 | “AI 시대라 필요” 같은 일반론으로 시작하지 않기 |
| 2. 질문 시장 | 구매/비교/문제해결/브랜드 질문셋 | 키워드 목록만 붙이지 않기 |
| 3. baseline | mention/source/citation/경쟁사 상태 | 단일 점수로 끝내지 않기 |
| 4. 실행 전략 | 콘텐츠/오프사이트/테크니컬 액션 | 실행 담당과 순서 명확히 하기 |
| 5. 리포트 방식 | 월간/분기 재측정과 의사결정 기준 | 예쁜 대시보드만 보여주지 않기 |
| 6. 일정/범위 | 4주/8주/월간 운영 계획 | 고객이 해야 할 일도 분리하기 |

## 제안서 한 장 요약 예시

```text
고객 문제: AI 답변에서 브랜드는 언급되지만 공식 페이지가 citation으로 잡히지 않는다.
핵심 질문군: 구매 비교 / 대안 탐색 / 가격/기능 / 리스크 검토
현재 상태: mention 있음, citation 약함, 경쟁사 source 우세
우선 액션: answer-first 페이지 3개 리라이트, FAQ/schema 보강, 외부 source 후보 5개 정리
측정 기준: 같은 질문셋 40개로 4주 후 mention/source/citation 변화 비교
운영 산출물: baseline 리포트, 콘텐츠 수정안, source map, 월간 리포트
```

## 사례로 이해하기

커머스/플랫폼형 고객은 상품 페이지가 많아도 AI 답변에서는 카테고리 설명, 비교표, 리뷰 출처, 가격 정책, 배송/반품 정보가 제대로 연결되지 않을 수 있습니다. 이때 제안서는 “상품 페이지를 더 많이 만들자”가 아니라 질문군별로 어떤 정보 단위가 부족한지 보여줘야 합니다.

PR/뉴스룸형 고객은 보도자료가 많아도 AI가 공식 뉴스룸보다 제3자 기사나 오래된 소개 페이지를 더 신뢰할 수 있습니다. 이 경우 제안서에는 뉴스룸 구조, 엔티티 일관성, source 후보 채널, 평판 리스크 질문군이 포함되어야 합니다.

## 실습 워크시트

| 입력 항목 | 작성 기준 |
|---|---|

| 질문 시장 | 구매/비교/문제해결/브랜드/리스크 질문군 |

| 재측정 기준 | 질문셋/기간/비교 지표 |

## 작성 예시

| 재측정 기준 | 같은 질문셋 30개를 4주 후 재측정 |

## 완료 기준

- 제안서가 고객 문제, 질문 시장, baseline, 실행 액션, 재측정을 한 흐름으로 설명합니다.
- 고객이 해야 할 일과 대행사/컨설턴트가 할 일이 분리되어 있습니다.
- HaloX 같은 분석 도구를 쓰는 이유가 기능 소개가 아니라 의사결정 기준으로 설명됩니다.

## 흔한 질문

**Q. 제안서에 가격을 먼저 넣어도 되나요?**

가격보다 범위가 먼저입니다. 질문 수, 리포트 깊이, 실행 범위, 재측정 주기가 정해져야 가격 근거가 생깁니다.

**Q. 제품 기능 화면을 많이 넣는 게 좋나요?**

화면은 필요하지만 중심은 아닙니다. 고객이 어떤 결정을 하게 되는지 설명하는 화면만 넣는 편이 좋습니다.
