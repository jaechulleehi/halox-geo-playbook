## 부록: HaloX 기능 연결 맵

이 부록은 앞 장에서 배운 GEO 개념을 HaloX의 기능적 흐름으로 연결하기 위한 안내서입니다. 공개 WikiDocs 본문은 교육 자료와 실무 워크북 역할을 하고, HaloX는 AI 검색 모니터링, 브랜드 가시성 분석, source/citation 추적, 실행 리포트 작성을 돕는 제품 레이어로 이어집니다.

핵심은 기능을 먼저 외우는 것이 아닙니다. 먼저 독자의 문제를 정리하고, 그 문제를 어떤 질문셋/지표/리포트/액션으로 바꿀지 정한 뒤, 필요한 HaloX 기능을 연결해야 합니다.

## 기능 연결의 기본 순서

HaloX와 WikiDocs를 함께 쓸 때는 아래 순서가 가장 자연스럽습니다.

1. WikiDocs로 개념과 판단 기준을 익힌다.
2. 자기 브랜드의 질문셋과 baseline을 만든다.
3. HaloX로 mention/source/citation/경쟁 문맥을 확인한다.
4. 콘텐츠/출처/기술/캠페인 액션으로 나눈다.
5. 30일 실행 리포트와 재측정 기준을 만든다.

이 순서를 지키면 HaloX는 단순 대시보드가 아니라 `진단 → 판단 → 실행 → 재측정`을 연결하는 운영 도구가 됩니다.

## 문제 유형별 기능 연결

같은 GEO라도 고객이 겪는 문제는 다릅니다. HaloX 기능은 문제 유형에 맞춰 연결해야 합니다.

| 문제 유형 | 먼저 물어볼 질문 | 연결 기능 | 다음 액션 |
|---|---|---|---|
| 브랜드가 AI 답변에 잘 안 나옴 | 어떤 질문군에서 빠지는가 | Mention map, 질문 포션 분석 | 추천형/비교형 질문셋 보강 |
| 출처로 쓰이지만 링크가 안 보임 | source와 citation이 분리되는가 | Source/Citation map | 인용 가능한 공식/외부 페이지 보강 |
| 경쟁사가 더 자주 추천됨 | 어떤 경쟁사와 함께 비교되는가 | Co-mention map, 경쟁 문맥 분석 | 비교 콘텐츠/포지셔닝 문구 작성 |
| 설명이 틀리거나 오래됨 | AI가 어떤 근거로 설명하는가 | Answer quality, Entity/평판 점검 | 소개문/FAQ/뉴스룸 구조 수정 |
| 상품/상세페이지가 읽히지 않음 | 상품 정보가 구조화되어 있는가 | 상세페이지 GEO 진단, schema/merchant feed 점검 | 상품 정보/리뷰/정책 구조화 |
| 캠페인 URL 성과가 불명확함 | URL이 반복 인용되는가 | Campaign URL citation tracking | 캠페인별 인용 안정성 리포트 |
| 지역/지점형 노출이 약함 | 지역 질문에서 후보로 들어가는가 | 지역 질문셋, 지점 URL/콘텐츠 클러스터 점검 | 지역 FAQ/리뷰/source 보강 |
| 대행사 제안을 검토해야 함 | 무엇을 증명하고 재측정하는가 | 대행사 검증 체크리스트, 리포트 평가 | 제안서 범위/지표/실행 액션 합의 |

## 강의에서 쓰는 방식

4주 강의에서는 HaloX 기능을 처음부터 모두 설명하지 않습니다. 매주 하나의 산출물에 필요한 기능만 보여주는 편이 좋습니다.

| 주차 | 수업 산출물 | 보여줄 HaloX 기능 | 주의할 점 |
|---|---|---|---|
| 1주차 | 브랜드 baseline 진단표 | 질문셋, mention/source/citation, 경쟁사 비교 | 점수 하나로 끝내지 않기 |
| 2주차 | Fan-out 질문맵과 콘텐츠 갭 | 질문 확장, 콘텐츠 갭, 질문 유형 분류 | 질문을 너무 넓게 만들지 않기 |
| 3주차 | 리라이트 초안과 source map | Answer-first 점검, source/citation 후보 | 자사 콘텐츠와 외부 출처를 섞지 않기 |
| 4주차 | 30일 실행 리포트 | 우선 액션, 재측정, 월간 리포트 | 실행 담당/기한/재측정 기준까지 남기기 |

강의의 목적은 기능 사용법을 암기시키는 것이 아니라, 수강생이 자기 브랜드의 GEO 운영표를 완성하게 만드는 것입니다.

## 세일즈와 제안서에서 쓰는 방식

세일즈 문맥에서는 기능 이름보다 고객 의사결정에 맞춰 설명해야 합니다.

| 제안서 섹션 | 보여줄 내용 | HaloX 연결 |
|---|---|---|
| 현재 문제 | AI 답변에서 빠지거나 잘못 설명되는 질문 | Baseline/mention map |
| 질문 시장 | 구매/비교/추천/검증형 질문셋 | 질문 포션/질문맵 |
| 근거 구조 | source/citation/외부 출처 상태 | Source/Citation map |
| 실행 액션 | 콘텐츠/출처/기술/캠페인 개선안 | 추천 액션 큐 |
| 재측정 | 4주 후 같은 질문셋으로 비교 | 월간 GEO 리포트 |

좋은 제안서는 “HaloX에 이런 기능이 있습니다”가 아니라 “이 고객은 어떤 질문에서 빠졌고, 무엇을 고치면 되고, 언제 다시 볼 것인가”를 보여줍니다.

## 내부 기능 스펙과 공개 문서의 경계

이 부록은 공개 문서이므로 실제 고객명, 계약 조건, 비공개 URL, 내부 화면 세부 사양은 다루지 않습니다. 대신 기능의 역할을 독자가 이해할 수 있는 수준으로 일반화합니다.

| 내부 기능 스펙의 언어 | 공개 문서에서의 표현 |
|---|---|
| 특정 고객용 진단 워크스페이스 | 산업별 GEO 진단표 |
| 세일즈 보조 화면 | 제안서/리포트 검증 체크리스트 |
| 상세페이지 진단 뷰 | 상품/페이지군 단위 GEO 점검 |
| 추천 액션 큐 | 다음 액션/30일 실행 플랜 |
| 운영 점수화 워크스페이스 | baseline/재측정/월간 리포트 |

공개 WikiDocs는 HaloX의 모든 내부 기능을 설명하는 제품 매뉴얼이 아닙니다. 독자가 GEO 문제를 이해하고, 그 문제를 HaloX에서 어떤 분석 흐름으로 볼 수 있는지 안내하는 연결 문서입니다.
